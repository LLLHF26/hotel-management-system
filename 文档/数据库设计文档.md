# 酒店管理系统 — 数据库设计文档


## 1. 概述

### 1.1 系统简介

本系统为酒店管理系统，采用 **微服务架构**，后端基于 Spring Cloud Alibaba，前端采用 Vue.js，AI 智能服务基于 Python FastAPI + LangChain。系统涵盖用户管理、客房管理、订单预订、财务管理、AI 智能客服与预警等业务模块。

### 1.2 数据库技术选型

| 技术 | 版本 | 用途 |
|------|------|------|
| MySQL | 8.0 | 核心业务数据库（4 个独立库） |
| SQLite | 3.x | AI 服务轻量数据存储 |
| ChromaDB | — | AI 知识库向量存储 |
| Redis | — | 缓存、会话管理 |

### 1.3 ORM / 持久层框架

| 服务 | 框架 |
|------|------|
| 后端微服务（Java） | MyBatis-Plus（逻辑删除、自动填充） |
| AI 服务（Python） | SQLAlchemy ORM |
| 连接池 | Alibaba Druid |

### 1.4 设计原则

- **每个微服务独立数据库**，避免跨库事务，降低耦合
- MySQL 统一使用 `utf8mb4` 字符集与 `InnoDB` 引擎
- 全部主键使用 `BIGINT AUTO_INCREMENT`
- 时间字段均使用 `DATETIME` 类型
- 通用字段（`create_time`、`update_time`、`is_deleted`）在各表中保持一致
- 冗余存储高频率跨服务查询字段（如 `room_number`、`customer_name`）

```
                  ┌──────────┐

           预订中◄─────┼─────►维修中
              │         │

          └───┬────┘    │
              │         │
              ▼         │
          ┌───────┐     │
          │ 打扫中 ├─────┘
          └───────┘
```

**状态流转规则**：
| 当前状态 | 可流转目标 |
|----------|----------|
| 空闲中 | 预订中、维修中 |
| 预订中 | 入住中、空闲中 |
| 入住中 | 待清洁中 |
| 待清洁中 | 打扫中 |
| 打扫中 | 空闲中 |
| 维修中 | 空闲中 |

## 6. 跨服务数据关系

由于微服务独立部署且各服务拥有独立数据库，无法使用数据库外键约束。跨库关系通过应用层维护。

| 关系 | 源表.字段 | 目标表.字段 | 所属服务 |
|------|------|------|------|
| 客户-订单 | orders.customer_id | customer.id | order → user |
| 房间-订单 | orders.room_id | room.id | order → room |
| 房型-房间 | room.room_type_id | room_type.id | room(内部) |
| 员工-清洁任务 | cleaning_task.cleaner_id | user.id | room → user |
| 房间-清洁任务 | cleaning_task.room_id | room.id | room(内部) |
| 房间-维修记录 | maintenance_record.room_id | room.id | room(内部) |
| 订单-额外消费 | order_extra.order_id | orders.id | order(内部) |
| 订单-支付记录 | payment.order_id | orders.id | order(内部) |
| 订单-退款记录 | refund_record.order_id | orders.id | finance → order |
| 支付-退款记录 | refund_record.payment_id | payment.id | finance → order |
| 员工-退款操作 | refund_record.operator_id | user.id | finance → user |
| 客户-积分流水 | points_log.customer_id | customer.id | user(内部) |
| 客户-对话日志 | chat_log.customer_id | customer.id | ai → user |

### 冗余字段策略

为减少跨服务 RPC 调用频率，以下字段做冗余存储：

| 表 | 冗余字段 | 来源 | 同步策略 |
|------|------|------|------|
| orders | customer_name, customer_phone | customer | 下单时写入，不随源头变更 |
| orders | room_number, room_type_name | room, room_type | 下单时写入，不随源头变更 |

| refund_record | order_no | orders.order_no | 退款时写入 |

| cleaning_task | cleaner_name | user.real_name | 创建任务时写入 |

## 7. 通用设计规范

### 7.1 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 数据库名 | 小写 + 下划线 | `hotel_user` |
| 表名 | 小写 + 下划线 | `room_type`, `points_log` |
| 字段名 | 小写 + 下划线 | `create_time`, `room_type_id` |
| 索引名 | 前缀 + 字段名 | `uk_username`, `idx_status` |
| 主键 | 统一 `id` | `id` |
| Java 实体 | 驼峰命名（Lombok） | `RoomType.java`, `PointsLog.java` |
| 布尔字段 | `is_` 前缀 | `is_deleted`, `is_enabled` |

### 7.2 数据类型规范

| 用途 | MySQL 类型 | 说明 |
|------|------|------|
| 主键ID | BIGINT | 统一自增 |
| 短字符串 | VARCHAR(N) | 精确限定长度 |
| 长文本 | TEXT | 适合 JSON 数组、对话内容 |
| 金额 | DECIMAL(10,2) / DECIMAL(12,2) | 避免浮点精度问题 |
| 日期 | DATE | 含月日即可 |
| 时间 | DATETIME | 需要时分秒 |
| 状态/枚举 | VARCHAR(16) | 中文值，便于可读 |
| 布尔标记 | TINYINT | 0/1 |
| 计数 | INT | 订单数、积分等 |

### 7.3 通用字段规范

每个核心业务表包含以下通用字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|------|------|
| create_time | DATETIME | CURRENT_TIMESTAMP | 由 `@TableField(fill = FieldFill.INSERT)` 自动填充 |
| update_time | DATETIME | CURRENT_TIMESTAMP ON UPDATE | 由 `@TableField(fill = FieldFill.INSERT_UPDATE)` 自动填充 |
| is_deleted | TINYINT | 0 | MyBatis-Plus 逻辑删除，`@TableLogic` |

### 7.4 其他约定

- 所有 MySQL 表使用 `InnoDB` 引擎
- 所有 MySQL 表字符集为 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`
- 枚举值在应用层通过枚举类管理（`common-service` 模块），数据库层面使用 `VARCHAR` 存储中文值，保证可读性
- MyBatis-Plus 配置：逻辑删除字段 `is_deleted`，已删除值 = 1，未删除值 = 0
- SQLite 数据库不使用逻辑删除，AI 服务各表独立管理生命周期
