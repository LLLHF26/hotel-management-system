# 酒店前台端（hotel-front-desk）界面文档

> 版本：v1（现状梳理）
> 说明：本文档如实记录**当前代码的真实情况**，包含每个界面的功能、调用的后端接口、数据来源，并标注存在的问题（重复页 / 空壳 / 逻辑不自洽 / 菜单缺失）。用于后续业务逻辑整改的依据。

---

## 一、总体结构

- 技术栈：Vue 3 + TypeScript + Vite + Element Plus + Pinia + Vue Router
- 布局：登录页独立；登录后进入 `FrontDeskLayout`（左侧深色导航 + 顶部 Header + 主内容区）
- 请求分两路：
  - 业务接口走 `request`（baseURL = `VITE_API_BASE`）→ Java 微服务
  - 告警接口走 `aiRequest`（baseURL = `/ai-api`）→ Python ai-service
- 响应约定：`code === 200` 时取 `data`

## 三、界面清单与功能说明

### 1. 登录页 `/login`
- **功能**：用户名 / 密码登录，登录后校验角色（仅 `FRONT_DESK` / `ADMIN` 可进），写入登录态并跳转工作台。
- **接口**：`login`（`POST /api/auth/login`）。
- **状态**：✅ 正常、独立。

### 2. 工作台（首页）`/front-desk/dashboard`
- **功能**：
  - 欢迎横幅（按时间显示上午好/下午好/晚上好）
  - 4 个统计卡片：今日入住 / 今日退房 / 今日预订 / 营业收入（含环比趋势）
  - 房态概览（房间网格 + 图例，"更多"跳房态管理）
  - 最新订单表格（前 5 条，"查看"跳订单详情）
  - 右侧：日历（纯前端）、快捷操作、系统通知时间线
- **接口**：`getRevenueSummary`（`/api/finance/revenue/summary`）、`getAlertStatus`（`/api/ai/alert/status`）、`getRoomDashboard`（`/api/room/dashboard`）、`getOrderList`（`/api/order/list`）。
- **数据来源**：全部真实接口，拉取失败优雅降级为 0 / 空。
- **状态**：✅ 正常。小瑕疵：欢迎横幅背景图为 Unsplash 外链；"系统通知—更多"是死链接（无点击事件）；快捷操作"办理入住/退房"都跳房态管理。

### 3. 房态管理 `/front-desk/room-status`
- **功能**：筛选（楼层/状态/关键字）→ 按楼层分组的房间卡片网格 → 房间详情抽屉（房型/楼层/状态/价格/保洁员）→ **办理入住**（打开新建订单弹窗，带入当前房间房型）/ **指派打扫**（选保洁员）。
- **接口**：`getRoomDashboard`、`getCleanerList`（`/api/user/cleaner/list`）、`assignCleaning`（`/api/room/{id}/cleaning/assign`）、`createOrder`（`/api/order/create`）、`checkIn`（`/api/order/{id}/check-in`，在办理入住弹窗中可选"立即入住"）。
- **状态**：✅ 功能最完整的房间界面，是唯一能"指派保洁"的入口；**本轮已把"办理入住"从死链接改为真实下单流程**（点空闲房 → 选房型/日期/住客 → 创建订单 → 可选立即入住）。

### 4. 入住办理 `/front-desk/checkin` —— 🗑️ 已删除（2026-07-20）
- 原为房态管理的删减拷贝且"办理入住"只跳转不生效。真实入住流程已并入**房态管理**，本路由与页面已移除。

### 5. 预订管理 `/front-desk/reservations` —— 🗑️ 已删除并合并（2026-07-20）
- 原为"订单管理"的逐行复制。已与订单管理合并为唯一的**订单中心**（`/front-desk/orders`）。

### 6. 订单中心 `/front-desk/orders`（原"订单管理"）
- **功能**：订单表格（筛选状态/日期/手机号）+ 操作（详情；待支付→支付/取消；已支付→入住；已入住→退房/续住/换房/消费）+ 顶部「新建预订 / 散客入住」按钮。
- **新建预订弹窗（`OrderCreateDialog` 组件）**：选房型（带房价）、入住/退房日期、会员识别（输入手机号 → `getCustomerList` 自动带出会员姓名/等级并关联 `customerId`，未匹配则按散客填姓名手机）、押金、备注、"立即入住"开关。提交时 `createOrder`（带 `X-Idempotency-Key` 防重复）→ 若"立即入住"则 `checkIn`。
- **接口**：`getOrderList`、`createOrder`、`checkIn`、`getRoomTypeList`（`/api/room/type/list`）、`getCustomerList`（`/api/customer/list`）。
- **状态**：✅ 本轮已重做，是原"预订管理 + 订单管理"合并后的唯一入口，并補齐了"新建预订/散客入住"这一此前缺失的核心能力。

### 7. 账单管理 `/front-desk/billing`
- **功能**：同样的订单表格，**操作最全**：待支付→支付/取消；已支付→入住；已入住→退房/续住/换房/消费；筛选栏含"散客入住""今日入住""今日退房"快捷按钮（与订单中心同源，作为前台操作台）。
- **接口**：`getOrderList`。
- **状态**：✅ 现与订单中心并存（订单中心偏查询，账单管理偏操作台）。两者同源同表属已知重叠，待后续将账单管理聚焦到财务视图（需接入 `finance/revenue/detail`、`/refunds`、`/report/export`，本轮未做）。

### 8. 退房办理 `/front-desk/checkout`
- **功能**：订单表格，**默认只列 `已入住` 订单**；操作：详情、退房、消费。
- **接口**：`getOrderList`（带 `status=已入住`）。
- **状态**：✅ 逻辑自洽（按订单退房），但实现仍是"订单列表"的又一个过滤变体，并非"按房间退房"。

### 9. 订单详情 `/front-desk/orders/:id`
- **功能**：订单信息 + 客户信息 + 房间信息 + 费用明细 + 支付记录；右侧快捷操作面板（按状态显示入住/退房/续住/换房/消费/收款/取消）。
- **接口**：`getOrderDetail`（`/api/order/{id}`），操作委托 `OrderActions` 组件。
- **状态**：✅ 正常、独立、信息完整。

### 10. 消息通知 `/front-desk/notifications`
- **功能**：筛选（全部/未读/已读）→ 告警时间线 → 单条已读 / 全部已读。
- **接口**：`getAlertStatus`、`markAlertAsRead`、`markAllAlertsAsRead`（`/api/ai/alert/*`）。
- **状态**：✅ 正常。与顶栏通知气泡同源，属合理的"完整页 + 弹层"拆分。

### 11. 系统设置 `/front-desk/settings`
- **功能**：个人资料（用户名/姓名/手机/邮箱/角色 + 保存）+ 修改密码。
- **接口**：`getProfile`、`updateProfile`、`changePassword`（`/api/user/*`）。
- **状态**：✅ 正常。

### 12. 个人信息 `/front-desk/profile` —— 🗑️ 已删除（2026-07-20）
- 原为"系统设置"的逐行复制。已删除本路由与页面，头像下拉"个人中心"现指向**系统设置**（`/front-desk/settings`）。
